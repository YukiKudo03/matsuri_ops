## 1. プロジェクト概要

### 1.1 背景
地域の祭りは、コミュニティの絆を深め、地域活性化に貢献する重要なイベントである。しかし、その運営には多岐にわたるタスク管理、予算管理、人員配置、当日の緊急対応など、複雑な調整が必要となる。

塩尻玄蕃まつりの運営経験を踏まえ、これらの課題を包括的に解決するWebアプリケーションを開発する。

### 1.2 目的
- 祭り運営の効率化と標準化
- リアルタイムな情報共有による迅速な意思決定
- 過去の知見の蓄積と継承
- 安全な祭り運営の実現

### 1.3 対象規模
| 規模 | 来場者数 | 出店数 |
|------|----------|--------|
| 小規模 | ~2,000人 | ~30店 |
| 中規模 | 2,000~10,000人 | 30~80店 |
| 大規模 | 10,000人~ | 80店~ |

## 3. 機能要件

### 3.1 認証・認可機能

#### FR-AUTH-001: ユーザー認証
- Magic Linkによるパスワードレス認証
- メールアドレスによるログイン
- セッション管理（有効期限設定）

#### FR-AUTH-002: ロールベースアクセス制御
- 8段階のロール管理
- ロール別の機能制限
- 動的な権限チェック

### 3.2 祭り管理機能

#### FR-FEST-001: 祭り情報管理
- 祭り基本情報の登録・編集・削除
- 開催期間、会場、規模の管理
- ステータス管理（企画中/準備中/開催中/終了/中止）

#### FR-FEST-002: メンバー管理
- 祭りへのメンバー割り当て
- 役割・担当エリアの設定
- メンバー一覧の表示

### 3.3 タスク管理機能

#### FR-TASK-001: タスクカテゴリ管理
11大タスクカテゴリの管理：
1. 企画・計画
2. 予算管理
3. 広報・PR
4. 会場設営
5. 交通・警備
6. 衛生・救護
7. 出店者管理
8. ボランティア管理
9. 法的手続き
10. 保険
11. 撤収・清掃

#### FR-TASK-002: タスク管理
- WBS構造（階層構造）でのタスク管理
- 親子関係によるサブタスク管理
- 優先度設定（低/中/高/緊急）
- ステータス管理（未着手/進行中/完了/ブロック/キャンセル）
- 進捗率の管理（0-100%）
- マイルストーン設定

#### FR-TASK-003: タスク依存関係
- タスク間の依存関係定義
- 依存タイプ（FS/SS/FF/SF）

#### FR-TASK-004: チェックリスト
- タスクに紐づくチェックリスト項目
- 完了/未完了の管理
- 完了者・完了日時の記録

### 3.4 予算・経費管理機能

#### FR-BUDG-001: 予算カテゴリ管理
- カテゴリ別の予算設定
- 予算額の登録・編集

#### FR-BUDG-002: 経費管理
- 経費の登録・編集・削除
- カテゴリ別の分類
- 支払方法の記録
- 領収書番号の管理
- 承認フロー（申請中→承認済→支払済）

#### FR-BUDG-003: 収入管理
- 収入源の登録
- 収入タイプ（協賛金/補助金/チケット売上/出店料/寄付/その他）
- ステータス管理（予定/確定/入金済/キャンセル）

#### FR-BUDG-004: 予実管理
- カテゴリ別の予算vs実績比較
- 残予算の自動計算
- 予算消化率の可視化

### 3.5 人員管理機能

#### FR-STAF-001: スタッフ情報管理
- スタッフの基本情報管理
- 役割・担当エリアの割り当て
- スキル情報の管理

#### FR-STAF-002: 統計表示
- 役割別の人数集計
- 配置状況の可視化

### 3.6 当日運営支援機能

#### FR-OPS-001: 運営ダッシュボード
- リアルタイム状況表示
- インシデント統計
- エリア混雑度表示

#### FR-OPS-002: インシデント管理
- インシデント報告・登録
- 重要度設定（低/中/高/緊急）
- カテゴリ分類（医療/警備/落とし物/天候/設備/その他）
- ステータス管理（報告済/確認済/対応中/解決済/クローズ）
- 担当者割り当て
- 対応内容の記録

#### FR-OPS-003: エリア状況管理
- エリア別混雑度（0-5段階）
- 気温・WBGT値の記録
- リアルタイム更新

#### FR-OPS-004: リアルタイム通知
- Phoenix PubSubによるリアルタイム更新
- インシデント発生時の即時通知
- エリア状況変更の即時反映

## 4. 非機能要件

### 4.1 性能要件

| 項目 | 要件 |
|------|------|
| 同時接続数 | 200ユーザー以上 |
| レスポンス時間 | 画面表示3秒以内 |
| リアルタイム更新 | 1秒以内 |
| 可用性 | 99.5%以上（当日は99.9%以上） |

### 4.2 セキュリティ要件

| 項目 | 要件 |
|------|------|
| 通信暗号化 | TLS 1.3 |
| パスワード | Bcryptによるハッシュ化 |
| セッション | セキュアCookie、CSRF対策 |
| アクセス制御 | ロールベース |
| 監査ログ | 重要操作の記録 |

### 4.3 ユーザビリティ要件

- レスポンシブデザイン（PC/タブレット/スマートフォン）
- 日本語UI
- 直感的なナビゲーション
- エラーメッセージの日本語表示

### 4.4 保守性要件

- モジュラー設計（Context/Schema分離）
- テストカバレッジ80%以上
- ドキュメント整備
- マイグレーションによるDB変更管理

## 5. 制約条件

### 5.1 技術制約
- Elixir/Phoenix Frameworkの使用
- PostgreSQLデータベース
- Phoenix LiveViewによるリアルタイムUI

### 5.2 運用制約
- クラウドホスティング（Fly.io または Gigalixir推奨）
- 日本国内からのアクセスが主

## 7. 用語定義

| 用語 | 定義 |
|------|------|
| 祭り（Festival） | 本システムで管理する1つのイベント単位 |
| タスク | 祭り運営に必要な作業項目 |
| インシデント | 当日発生する問題・緊急事態 |
| WBS | Work Breakdown Structure（作業分解構成図） |
| WBGT | 暑さ指数（Wet Bulb Globe Temperature） |
| PubSub | Publish-Subscribe（発行-購読型メッセージング） |
